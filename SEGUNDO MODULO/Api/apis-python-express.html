Fundamento 01

¿Qué es una API?

API significa Application Programming Interface — Interfaz de Programación de Aplicaciones. En términos simples: es un programa que escucha peticiones y devuelve respuestas. No tiene pantalla, no tiene botones. Solo recibe datos y responde datos.

Analogía: El mesero del restaurante

Tú (el cliente) le dices al mesero (la API) qué quieres. El mesero va a la cocina (la base de datos / lógica), consigue lo que pediste, y te lo trae de vuelta. Tú nunca entras a la cocina. El mesero es el intermediario.

Las APIs son la base de todo internet moderno. Cuando abres Twitter, una app en tu celular le hace peticiones a una API de Twitter que devuelve los tweets en formato JSON. El celular se encarga de mostrarlos bonito.

¿Cómo se "habla" con una API?

A través del protocolo HTTP, el mismo que usas cuando navegas en Chrome. Una petición tiene:

HTTP Request 
GET /usuarios/42 HTTP/1.1
Host: api.miapp.com
Content-Type: application/json

Y la API responde:

HTTP Response 
200 OK
Content-Type: application/json

{
  "id": 42,
  "nombre": "Gabriel Niño",
  "rol": "CEO"
}
Fundamento 02

App Estática vs App con API

Esta es una diferencia clave que confunde a muchos principiantes. ¿Por qué necesito un servidor corriendo si ya tengo un archivo HTML?

🗂️ App Estática
  • Son archivos: .html, .css, .js
  • Viven en un disco o CDN
  • No tienen lógica del lado del servidor
  • Todos ven exactamente lo mismo
  • No leen bases de datos
  • No se puede "iniciar sesión" sin una API
  • Ejemplo: currículum en línea, landing page
⚙️ App con API (Dinámica)
  • Hay un proceso corriendo en un servidor
  • Responde diferente según quien pregunta
  • Puede conectarse a bases de datos
  • Puede autenticar usuarios
  • Puede recibir y guardar datos
  • Necesita un puerto abierto (ej: :3000)
  • Ejemplo: Twitter, Netflix, WhatsApp
⚠️ Diferencia clave
Un archivo HTML estático no escucha. Una API sí escucha. Es como la diferencia entre un letrero pegado en la pared (estático) y un recepcionista sentado en un escritorio (API) que responde preguntas en tiempo real.
Característica Sitio Estático API / App Dinámica
¿Necesita proceso corriendo? ❌ No ✅ Sí, siempre
¿Lee base de datos? ❌ No ✅ Sí
¿Respuesta personalizada? ❌ Igual para todos ✅ Distinta por usuario
¿Puerto abierto? ❌ No necesario ✅ Obligatorio
¿Consume memoria RAM? Casi nada Sí, mientras está activa
Fundamento 03

¿Qué significa que una API esté "arriba" o "encendida"?

Cuando dices que una API está "arriba", significa que hay un proceso activo en un servidor que está constantemente escuchando en un puerto específico, esperando que alguien haga una petición.

API Corriendo — proceso activo escuchando en localhost:5000
API Apagada — nadie escucha, cualquier petición dará error de conexión

El concepto de Puerto (Port)

Un puerto es como una "puerta numerada" en tu computador. Hay 65,535 puertas posibles. Cuando dices localhost:5000, estás diciendo: "quiero entrar por la puerta número 5000 de esta máquina". Si nadie está "parado" en esa puerta, la conexión falla.

Analogía: El edificio de apartamentos

La dirección IP es el edificio. El puerto es el número de apartamento. Si vas al apartamento 5000 y nadie vive ahí (la API está apagada), nadie te abre. Si la API está corriendo, hay alguien ahí que contesta.

¿Por qué hay que tenerla "encendida" siempre?

Porque HTTP es sincrono y sin estado. El cliente hace una petición, la API responde, y la conexión se cierra. Pero el proceso que gestiona esas conexiones debe existir permanentemente para estar listo cuando llegue la próxima petición. Si apagas el proceso, nadie puede hacer peticiones.

Terminal — API encendida 
# Cuando inicias la API, ves algo así:
$ python app.py

 * Running on http://127.0.0.1:5000
 * Debug mode: on
# ↑ Esto significa: "Estoy escuchando. Puedes hablarme."
# El proceso NO termina. Se queda esperando peticiones.
🔌 En producción
En un servidor real (AWS, DigitalOcean, etc.) la API vive en un proceso manejado por herramientas como pm2, gunicorn, o systemd, que la reinician automáticamente si se cae. Así garantizan que siempre esté "arriba".
Python / Flask — 01

Tu primera API en Python con Flask

Flask es un microframework de Python para crear APIs web. "Micro" no significa pequeño, significa que viene con lo esencial y tú agregas lo que necesites.

Instalar Flask

Terminal 
pip install flask

La API más pequeña del mundo

Python — app.py 
from flask import Flask

app = Flask(__name__)     # 1. Crear la aplicación

@app.route('/')           # 2. Definir un endpoint en la ruta "/"
def inicio():             # 3. Función que maneja esa ruta
    return '¡Hola, mundo!'  # 4. Lo que devuelve

if __name__ == '__main__':
    app.run(debug=True)     # 5. Encender el servidor

Ejecutar la API

Terminal 
$ python app.py

 * Running on http://127.0.0.1:5000
 * Debug mode: on

Ahora puedes abrir tu navegador en http://localhost:5000 y verás "¡Hola, mundo!". Tu API está arriba y respondiendo.

Python / Flask — 02

¿Qué es un Endpoint?

Un endpoint es una URL específica que tu API expone y que hace algo concreto cuando la llaman. Es el punto de contacto entre el cliente y tu lógica.

Analogía: Los cajeros del banco

El banco (tu API) tiene varios cajeros (endpoints). El cajero de la ventanilla 1 (/depositar) recibe depósitos. El de la ventanilla 2 (/retirar) da dinero. El de la 3 (/saldo) informa tu saldo. Cada ventanilla tiene una función específica.

Múltiples endpoints en Flask

Python — app.py 
from flask import Flask, jsonify

app = Flask(__name__)

# Endpoint 1: ruta raíz
@app.route('/')
def inicio():
    return jsonify({ 'mensaje': 'API funcionando' })

# Endpoint 2: lista de usuarios
@app.route('/usuarios')
def obtener_usuarios():
    usuarios = [
        { 'id': 1, 'nombre': 'Ana' },
        { 'id': 2, 'nombre': 'Luis' }
    ]
    return jsonify(usuarios)

# Endpoint 3: un usuario por ID (parámetro dinámico)
@app.route('/usuarios/<int:id>')
def obtener_usuario(id):
    return jsonify({ 'id': id, 'nombre': 'Ana' })

Ahora tu API tiene 3 endpoints distintos. Cada URL hace algo diferente:

GET / GET /usuarios GET /usuarios/1 GET /usuarios/42
🐍 El decorador @app.route
La línea @app.route('/usuarios') es un decorador de Python. Le dice a Flask: "registra la función que viene abajo para que se ejecute cuando alguien haga una petición a /usuarios". Es el mecanismo que conecta una URL con una función.
Python / Flask — 03

¿Qué es un Controlador?

Un controlador es la función (o clase) que se encarga de manejar la petición: recibir los datos que llegaron, llamar a la lógica de negocio (o base de datos), y preparar la respuesta. Es el "cerebro" que decide qué hacer con cada petición.

En Flask básico, el controlador es directamente la función decorada con @app.route. En proyectos más grandes, los controladores viven en archivos separados.

Estructura sin separar (principiante)

Python — todo en un archivo 
@app.route('/productos')
def listar_productos():          # ← Este ES el controlador
    # 1. Obtener datos (simulando base de datos)
    productos = [
        {'id': 1, 'nombre': 'Laptop', 'precio': 2500000},
        {'id': 2, 'nombre': 'Mouse',  'precio': 45000},
    ]
    # 2. Preparar y retornar la respuesta
    return jsonify(productos)

Estructura separada (recomendada en proyectos reales)

Estructura de archivos 
mi_api/
├── app.py                  # Punto de entrada, define la app
├── routes/
│   └── productos_routes.py # Define las rutas (URLs)
└── controllers/
    └── productos_ctrl.py   # Contiene la lógica del controlador
controllers/productos_ctrl.py 
from flask import jsonify, request

# El controlador: solo lógica, sin saber de rutas
def obtener_todos():
    productos = [
        {'id': 1, 'nombre': 'Laptop'},
        {'id': 2, 'nombre': 'Mouse'},
    ]
    return jsonify(productos), 200

def crear_producto():
    datos = request.get_json()       # Leer el body de la petición
    nombre = datos.get('nombre')
    # Aquí iría: guardar en base de datos
    return jsonify({'mensaje': f'Producto {nombre} creado'}), 201
💡 Regla de oro
La ruta define el "dónde" (la URL). El controlador define el "qué" (la lógica). Separar estas responsabilidades hace tu código más fácil de mantener y testear.
Python / Flask — 04

Flujo completo de una petición

Cuando alguien hace GET /usuarios/5 a tu API en Flask, esto es exactamente lo que ocurre, paso a paso:

1

El cliente hace la petición

Un navegador, Postman, o una app hace GET http://localhost:5000/usuarios/5. Esto genera un paquete HTTP que viaja por la red.

2

Flask recibe en el puerto 5000

El proceso de Flask, que está escuchando en el puerto 5000, detecta la petición entrante. El servidor WSGI (Werkzeug en desarrollo) la captura.

3

Router: encontrar el endpoint correcto

Flask revisa su tabla de rutas registradas. Ve que /usuarios/<int:id> coincide con /usuarios/5. Extrae el parámetro id = 5.

4

Se ejecuta el controlador

Flask llama a la función obtener_usuario(id=5). Aquí vive tu lógica: puede consultar una base de datos, hacer cálculos, etc.

5

Se prepara la respuesta

El controlador retorna jsonify({...}). Flask convierte ese dict de Python en texto JSON y arma el paquete HTTP de respuesta con código 200 OK.

6

Respuesta enviada al cliente

El paquete viaja de vuelta. El cliente recibe el JSON. La conexión se cierra. Flask sigue escuchando para la próxima petición.

Python — Ejemplo completo del flujo 
from flask import Flask, jsonify

app = Flask(__name__)

# Simulamos una "base de datos" en memoria
usuarios_db = {
    1: {'nombre': 'Ana García',  'email': 'ana@email.com'},
    2: {'nombre': 'Luis Mora',   'email': 'luis@email.com'},
    5: {'nombre': 'Sofía Ruiz',  'email': 'sofia@email.com'},
}

@app.route('/usuarios/<int:id>')
def obtener_usuario(id):
    usuario = usuarios_db.get(id)       # Buscar en la "BD"

    if not usuario:                      # Si no existe...
        return jsonify({
            'error': 'Usuario no encontrado'
        }), 404                          # ...código 404

    return jsonify({                     # Si existe...
        'id': id,
        **usuario
    }), 200                             # ...código 200 OK

if __name__ == '__main__':
    app.run(debug=True)
Python / Flask — 05

GET, POST, PUT, DELETE

HTTP define verbos (métodos) que indican la intención de la petición. No es solo una convención: es un estándar que todos entienden.

Método Significado Analogía Código típico
GET Obtener / Leer Consultar el menú del restaurante 200 OK
POST Crear algo nuevo Hacer un pedido nuevo 201 Created
PUT Reemplazar completo Cambiar todo el pedido 200 OK
PATCH Modificar parcial Cambiar solo el postre 200 OK
DELETE Eliminar Cancelar el pedido 204 No Content
Python — CRUD completo de productos 
from flask import Flask, jsonify, request

app = Flask(__name__)
productos = []

# GET — listar todos
@app.route('/productos', methods=['GET'])
def listar():
    return jsonify(productos), 200

# POST — crear uno nuevo
@app.route('/productos', methods=['POST'])
def crear():
    datos = request.get_json()          # Leer JSON del body
    nuevo = {
        'id': len(productos) + 1,
        'nombre': datos['nombre'],
        'precio': datos['precio']
    }
    productos.append(nuevo)
    return jsonify(nuevo), 201          # 201 = Creado

# DELETE — eliminar por ID
@app.route('/productos/<int:id>', methods=['DELETE'])
def eliminar(id):
    global productos
    productos = [p for p in productos if p['id'] != id]
    return '', 204                       # 204 = Sin contenido
Express / Node.js — 01

Tu primera API en Express

Express es un framework minimalista para Node.js. Node.js te permite ejecutar JavaScript fuera del navegador — en el servidor. Express agrega el manejo de rutas y peticiones HTTP encima de Node.

Instalar Node.js y Express

Terminal 
# Crear proyecto e instalar Express
npm init -y
npm install express

La API más pequeña en Express

JavaScript — index.js 
const express = require('express')   // 1. Importar Express
const app     = express()              // 2. Crear la aplicación

app.get('/', (req, res) => {         // 3. Definir endpoint GET /
    res.json({ mensaje: '¡Hola!' })   // 4. Responder con JSON
})

app.listen(3000, () => {              // 5. Encender en puerto 3000
    console.log('API en puerto 3000')
})

Ejecutar

Terminal 
$ node index.js
API en puerto 3000
# El proceso no termina → está escuchando
⚡ req y res
En Express, cada handler recibe dos objetos: req (request = la petición que llegó, con todos sus datos) y res (response = el objeto con el que mandas tu respuesta). Todo gira alrededor de estos dos.
Express / Node.js — 02

Endpoints en Express

En Express, los endpoints se definen con app.get(), app.post(), app.put(), app.delete(), etc. El primer argumento es la ruta, el segundo es el handler (la función que maneja la petición).

JavaScript — múltiples endpoints 
const express = require('express')
const app = express()
app.use(express.json())              // Middleware: parsear JSON

// Endpoint 1: raíz
app.get('/', (req, res) => {
    res.json({ status: 'ok' })
})

// Endpoint 2: lista de usuarios
app.get('/usuarios', (req, res) => {
    res.json([
        { id: 1, nombre: 'Ana' },
        { id: 2, nombre: 'Luis' }
    ])
})

// Endpoint 3: usuario por ID (parámetro dinámico)
app.get('/usuarios/:id', (req, res) => {
    const { id } = req.params        // Extraer :id de la URL
    res.json({ id: parseInt(id), nombre: 'Ana' })
})

app.listen(3000)

Nota el :id en Express — equivale al <int:id> de Flask. Es el parámetro dinámico de la URL, accesible en req.params.id.

Express / Node.js — 03

Controladores en Express

Al igual que en Flask, en Express puedes (y debes) separar los controladores en archivos propios. La ruta y la lógica van separadas.

Estructura de proyecto 
mi-api/
├── index.js                    # Punto de entrada
├── routes/
│   └── productosRoutes.js      # Define las rutas
└── controllers/
    └── productosController.js  # Contiene la lógica
controllers/productosController.js 
const productos = [
    { id: 1, nombre: 'Laptop', precio: 2500000 },
    { id: 2, nombre: 'Mouse',  precio: 45000  },
]

// Controlador: obtener todos los productos
const obtenerTodos = (req, res) => {
    res.status(200).json(productos)
}

// Controlador: crear un producto
const crearProducto = (req, res) => {
    const { nombre, precio } = req.body    // Datos del body
    const nuevo = {
        id: productos.length + 1,
        nombre,
        precio
    }
    productos.push(nuevo)
    res.status(201).json(nuevo)
}

module.exports = { obtenerTodos, crearProducto }
routes/productosRoutes.js 
const express    = require('express')
const router     = express.Router()
const ctrl       = require('../controllers/productosController')

router.get('/',  ctrl.obtenerTodos)   // GET  /productos
router.post('/', ctrl.crearProducto)  // POST /productos

module.exports = router
index.js — punto de entrada 
const express         = require('express')
const productosRoutes = require('./routes/productosRoutes')

const app = express()
app.use(express.json())

// Montar las rutas bajo el prefijo /productos
app.use('/productos', productosRoutes)

app.listen(3000, () => console.log('🚀 Servidor en :3000'))
Express / Node.js — 04

Flujo de una petición en Express

El flujo es similar a Flask pero con una pieza extra importante: el Middleware. En Express, las peticiones pasan por una cadena de middleware antes de llegar al controlador.

1

Cliente hace POST /productos con body JSON

La petición incluye un body: {"nombre": "Teclado", "precio": 120000}

2

Express recibe en el puerto 3000

Node.js tiene su propio servidor HTTP. Express lo usa. La petición llega al proceso activo.

3

Middleware: express.json()

Antes de que llegue al controlador, el middleware parsea el body de texto JSON a un objeto JavaScript. Sin esto, req.body estaría vacío.

4

Router encuentra la ruta correcta

Express ve que POST /productos coincide con router.post('/') montado en /productos.

5

Se ejecuta el controlador crearProducto

La función recibe req (con req.body.nombre y req.body.precio) y res para responder.

6

Respuesta 201 Created con el nuevo objeto

El cliente recibe el JSON del producto creado. Express continúa escuchando nuevas peticiones.

🔗 ¿Qué es Middleware?
Un middleware es una función que se ejecuta entre que llega la petición y que llega al controlador. Puede leer/modificar req y res, o decidir si continúa o no. express.json() es un middleware que convierte el body de texto a objeto JS. Puedes tener tantos como necesites: logging, autenticación, validación, etc.

Glosario

Conceptos esenciales

📌 API

Programa que escucha peticiones HTTP y devuelve respuestas. No tiene interfaz visual. Habla en JSON.

📌 Endpoint

Una URL específica que tu API expone. Ejemplo: /usuarios, /productos/5. Cada endpoint hace exactamente una cosa.

📌 Controlador

La función que maneja lo que sucede cuando llega una petición a un endpoint. Contiene la lógica de negocio.

📌 Puerto (Port)

Canal numérico por donde entra el tráfico. 5000 para Flask, 3000 para Express. La API "escucha" en ese puerto.

📌 HTTP Method (Verbo)

La intención de la petición: GET (leer), POST (crear), PUT (reemplazar), PATCH (modificar), DELETE (eliminar).

📌 JSON

JavaScript Object Notation. El formato estándar en que las APIs devuelven datos. Es texto plano que representa objetos y listas.

📌 Middleware

Función que procesa la petición antes de que llegue al controlador. Puede parsear JSON, verificar autenticación, registrar logs, etc.

📌 Request (req)

El objeto que contiene todo sobre la petición entrante: URL, método, headers, body, parámetros, cookies.

📌 Response (res)

El objeto con el que construyes y envías tu respuesta al cliente: código de estado, headers, cuerpo.

📌 Código de estado HTTP

200 OK · 201 Creado · 400 Petición inválida · 404 No encontrado · 500 Error del servidor.

📌 localhost

Dirección que apunta a tu propia máquina. localhost:5000 = "en este computador, puerto 5000".
