Módulo 1 · 30 min

¿Qué es Git y por qué existe?

⏱ 0:00 — 0:30

El problema que Git resuelve

Imagina que llevas semanas desarrollando un sitio web. De repente, algo se rompe y no recuerdas qué cambiaste. O peor: dos compañeros editan el mismo archivo y uno sobreescribe el trabajo del otro.

Eso es exactamente lo que pasaba antes de que existieran herramientas de control de versiones. La gente hacía esto:

EL INFIERNO SIN GIT 
📁 mi-proyecto/
   ├── index.html
   ├── index-final.html
   ├── index-final-v2.html
   ├── index-final-v2-ESTE.html
   ├── index-BUENO.html
   └── index-ESTE-SI-FUNCIONA.html   ← 😱
💡 Analogía

Git es como el historial de cambios de Google Docs, pero para cualquier tipo de archivo o proyecto. Guarda una "foto" de tu proyecto cada vez que tú decides, y puedes regresar a cualquier momento en el pasado.

Git vs GitHub — No son lo mismo

Este es el error más común de los principiantes. Son dos cosas distintas:

Característica Git GitHub
¿Qué es? Herramienta de software Plataforma web (empresa)
¿Dónde vive? En tu computadora En internet (la nube)
¿Creado por? Linus Torvalds (2005) Tom Preston-Werner (2008)
¿Necesita internet? No Sí
¿Costo? Gratuito y open source Gratis con planes pagos
Alternativas SVN, Mercurial GitLab, Bitbucket
💡 Analogía

Git es como Microsoft Word (el programa). GitHub es como Google Drive (el lugar donde guardas y compartes tus archivos de Word). Puedes usar Word sin Drive, pero juntos son más poderosos.

Conceptos clave que usarás hoy

📁

Repositorio (repo)

Es la carpeta de tu proyecto con todo su historial de cambios. Puede ser local (en tu PC) o remoto (en GitHub).

📸

Commit

Es una "foto" de tu proyecto en un momento dado. Cada commit tiene un mensaje que describe qué cambiaste.

🌿

Branch (rama)

Una línea de desarrollo independiente. Puedes trabajar en una nueva función sin afectar el código principal.

🔀

Merge

Combinar los cambios de una rama con otra. Es como unir dos versiones del mismo documento.

⬆️

Push

Enviar tus commits locales al repositorio remoto (GitHub). "Subir" tu trabajo.

⬇️

Pull

Descargar los cambios del repositorio remoto a tu máquina local. "Bajar" el trabajo de otros.

Instalación y configuración inicial

Antes de empezar, verifica si Git ya está instalado en tu computadora:

TERMINAL / BASH 
# Verificar si Git está instalado
git --version

# Respuesta esperada: git version 2.x.x
📥 Instalación

Si no tienes Git instalado: ve a git-scm.com → Download → ejecuta el instalador. En Mac usa brew install git. En Ubuntu: sudo apt install git

Ahora configura tu identidad. Git necesita saber quién hace cada commit:

CONFIGURACIÓN GLOBAL — SOLO SE HACE UNA VEZ 
# Tu nombre (aparecerá en cada commit)
git config --global user.name "Tu Nombre Aquí"

# Tu correo (debe ser el mismo de tu cuenta GitHub)
git config --global user.email "tucorreo@ejemplo.com"

# Editor por defecto (recomendado: VS Code)
git config --global core.editor "code --wait"

# Verificar que todo quedó bien
git config --list
⚠️ Importante

El correo que configures aquí debe coincidir exactamente con el correo de tu cuenta de GitHub. De lo contrario, tus commits no quedarán vinculados a tu perfil.

⚡ Ejercicio 1
Configura Git en tu máquina
⏱ 5 min
🎯 Objetivo: Instalar Git (si no lo tienes) y configurar tu identidad para que todos tus futuros commits lleven tu nombre.
  1. Abre tu terminal Windows: busca "Git Bash" o "PowerShell". Mac/Linux: Terminal normal.
  2. Verifica la instalación con git --version Debes ver algo como "git version 2.43.0"
  3. Configura tu nombre: git config --global user.name "Tu Nombre" Usa tu nombre real tal como aparece en GitHub o como lo quieras ver en los commits.
  4. Configura tu correo: git config --global user.email "correo@..." Usa el mismo correo de tu cuenta de GitHub.
  5. Verifica con git config --list Confirma que aparece tu nombre y correo correctamente.

Módulo 2 · 30 min

Tu primer repositorio local

⏱ 0:30 — 1:00

El flujo básico de Git

Antes de ver comandos, entiende el flujo. En Git hay tres "zonas" donde viven tus archivos:

  WORKING DIRECTORY       STAGING AREA          REPOSITORIO (commits)
  ┌─────────────────┐     ┌──────────────────┐    ┌──────────────────────┐
  │                 │     │                  │    │                      │
  │  Archivos que   │────▶│  Archivos listos │───▶│  Historial de commits│
  │  estás editando │     │  para el commit  │    │  (permanente)        │
  │                 │     │                  │    │                      │
  └─────────────────┘     └──────────────────┘    └──────────────────────┘
         ↑                      ↑                          ↑
    Donde trabajas        git add                git commit
💡 Analogía del supermercado

Piensa en hacer mercado: el Working Directory es tu carrito (donde vas echando cosas), el Staging Area es la banda transportadora (los artículos que decidiste comprar), y el commit es el tiquete de compra (el registro permanente de lo que llevaste).

Comandos fundamentales

INICIALIZAR UN REPOSITORIO 
# 1. Crear una carpeta nueva y entrar en ella
mkdir mi-primer-repo
cd mi-primer-repo

# 2. Inicializar Git en esa carpeta
git init

# Respuesta: Initialized empty Git repository in .../mi-primer-repo/.git/

# Ahora existe una carpeta oculta .git — ¡ahí vive toda la magia!
VER EL ESTADO DEL REPOSITORIO 
# Este comando es tu mejor amigo — úsalo TODO el tiempo
git status

# Sin archivos:
# On branch main
# No commits yet
# nothing to commit

# Después de crear un archivo:
# Untracked files:
#   (use "git add <file>..." to include in what will be committed)
#         index.html
AÑADIR ARCHIVOS AL STAGING AREA 
# Añadir un archivo específico
git add index.html

# Añadir múltiples archivos
git add index.html style.css

# Añadir TODOS los archivos modificados (el más común)
git add .

# Verificar qué quedó en staging
git status
# Changes to be committed:
#       new file: index.html
HACER UN COMMIT 
# Commit con mensaje (forma más común)
git commit -m "Agrega estructura inicial del HTML"

# [main (root-commit) a1b2c3d] Agrega estructura inicial del HTML
#  1 file changed, 12 insertions(+)

# Añadir al staging Y hacer commit en un solo paso (solo archivos ya tracked)
git commit -am "Actualiza el título de la página"
💬 Buenos mensajes de commit

Un buen mensaje describe QUÉ cambiaste y POR QUÉ, no el cómo. ✅ "Corrige error de validación en formulario de contacto" — ❌ "cambios", "fix", "asdfg"

VER EL HISTORIAL DE COMMITS 
# Ver historial completo
git log

# Vista resumida (la más usada en el día a día)
git log --oneline
# a1b2c3d Agrega estilos de la página principal
# 9f8e7d6 Corrige error en el formulario
# 3c2b1a0 Agrega estructura inicial del HTML

# Ver con gráfico de ramas
git log --oneline --graph --all
VER DIFERENCIAS ENTRE VERSIONES 
# Ver qué cambié en archivos no añadidos aún
git diff

# Ver qué hay en staging (lo que va en el próximo commit)
git diff --staged

# Comparar dos commits específicos
git diff a1b2c3d 9f8e7d6

Deshacer cambios — No entres en pánico

Una de las grandes ventajas de Git es que casi siempre puedes deshacer errores:

DESHACER CAMBIOS 
# Quitar un archivo del staging (sin perder los cambios)
git restore --staged index.html

# Descartar cambios en un archivo (vuelve al último commit)
git restore index.html

# Deshacer el último commit (mantiene los cambios en staging)
git reset --soft HEAD~1

# Corregir el mensaje del último commit
git commit --amend -m "Mensaje corregido"
🔒 Regla de oro

Nunca uses git reset --hard si no sabes exactamente qué hace. Es el comando más peligroso para principiantes porque elimina cambios de forma permanente. Cuando tengas dudas, usa git status para entender dónde estás.

⚡ Ejercicio 2
Tu primera sesión de Git local
⏱ 15 min
🎯 Objetivo: Crear un repositorio local, hacer al menos 3 commits y practicar el ciclo completo: editar → add → commit → log.
  1. Crea la carpeta y entra en ella mkdir mi-sitio-web y luego cd mi-sitio-web
  2. Inicializa el repositorio con git init Verifica que aparezca el mensaje de inicialización.
  3. Crea un archivo index.html con estructura básica Puedes abrirlo con VS Code: code index.html. Escribe un HTML simple con un título y un párrafo.
  4. Haz git status — ¿qué ves? Debes ver index.html en rojo como "untracked file".
  5. Añade el archivo: git add index.html y confirma con git status Ahora debería aparecer en verde como "new file".
  6. Haz tu primer commit: git commit -m "Crea página principal con estructura HTML" ¡Felicitaciones! Acabas de hacer tu primer commit. 🎉
  7. Modifica el archivo, añade un CSS, haz 2 commits más Practica el ciclo completo. Verifica el historial con git log --oneline.

Módulo 3 · 30 min

Conectando con GitHub

⏱ 1:00 — 1:30

Crear tu cuenta en GitHub

Si aún no tienes cuenta, ve a github.com y regístrate. Elige un nombre de usuario profesional — ese nombre aparecerá en tus proyectos y en tu perfil público.

🏠

Perfil público

Tu GitHub es como tu portafolio profesional. Los empleadores lo revisan. Cuídalo.

⭐

Stars y forks

Puedes "marcar" repos que te gustan con estrella, o "forkearlos" para hacer tu propia copia.

🔍

Explorar repos

GitHub tiene millones de proyectos open source. Todo lo que usas tiene código aquí.

Crear un repositorio en GitHub

Ahora vamos a crear el repositorio remoto en GitHub para subir tu proyecto local:

1. Click en el botón "+" → "New repository"

Está en la esquina superior derecha de GitHub.

2. Dale un nombre al repositorio

Usa el mismo nombre de tu carpeta local. Sin espacios, sin tildes. Ejemplo: mi-sitio-web

3. Elige Public o Private

Public: cualquiera puede verlo. Private: solo tú y colaboradores que invites.

4. NO marques "Initialize this repository"

Porque ya tienes commits locales. Si lo inicializas en GitHub también, habrá conflicto.

5. Click en "Create repository"

GitHub te mostrará instrucciones. ¡Ya tienes tu repositorio remoto!

Conectar tu repo local con GitHub

GitHub te dará comandos exactos al crear el repo. Así es como funciona:

CONECTAR Y SUBIR POR PRIMERA VEZ 
# 1. Agregar el remoto (tu repo de GitHub como destino)
git remote add origin https://github.com/tu-usuario/mi-sitio-web.git

# 2. Verificar que quedó bien
git remote -v
# origin  https://github.com/tu-usuario/mi-sitio-web.git (fetch)
# origin  https://github.com/tu-usuario/mi-sitio-web.git (push)

# 3. Subir tu código por primera vez
git push -u origin main

# Enumerating objects: 6, done.
# Writing objects: 100% (6/6), 1.23 KiB | 1.23 MiB/s, done.
# To https://github.com/tu-usuario/mi-sitio-web.git
#  * [new branch]      main -> main
🔑 Autenticación

La primera vez que hagas push, GitHub puede pedirte autenticarte. La forma moderna es usar un Personal Access Token (PAT): ve a Settings → Developer Settings → Personal Access Tokens → Generate new token. Úsalo como contraseña cuando te lo pida.

Flujo diario con GitHub

Una vez conectado, tu flujo de trabajo diario será este:

FLUJO DE TRABAJO DIARIO 
# Al inicio del día: bajar cambios del equipo
git pull origin main

# ... trabajas, editas archivos ...

# Ver qué cambiaste
git status

# Añadir cambios
git add .

# Guardar con descripción
git commit -m "Implementa sección de contacto con validación"

# Subir a GitHub
git push origin main

# ¡Listo! Tu trabajo está en la nube 🚀
CLONAR UN REPOSITORIO EXISTENTE 
# Clonar un proyecto de GitHub a tu PC (crea la carpeta automáticamente)
git clone https://github.com/usuario/nombre-repo.git

# Entrar a la carpeta clonada
cd nombre-repo

# Ya tienes todo: archivos + historial completo
git log --oneline
  TU PC (LOCAL)                             GITHUB (REMOTO)

  ┌────────────────────┐                    ┌────────────────────┐
  │                    │   git push ──────▶ │                    │
  │   Working Dir      │                    │   Repositorio      │
  │   + Commits        │   ◀────── git pull │   Remoto           │
  │                    │                    │   (origin)         │
  └────────────────────┘                    └────────────────────┘
                                                     │
                              git clone ─────────────┘
                              (para empezar desde cero en otra PC)

Módulo 4 · 30 min

Ramas y Pull Requests

⏱ 1:30 — 2:00

¿Por qué usar ramas?

Imagina que tu sitio web ya está en producción y funciona perfecto. Tu jefe te pide agregar una nueva sección. No quieres tocar el código que ya funciona mientras experimentas. Para eso existen las ramas.

💡 Analogía

Las ramas son como universos paralelos de tu proyecto. En el universo principal (main) todo funciona. En tu rama nueva (feature/nueva-seccion) puedes experimentar sin miedo. Cuando termines y todo esté bien, "fusionas" (merge) esos universos.

                                          feature/nueva-seccion
                                         ●─────●─────●
                                        /               \
  main   ●─────●─────●─────────────────●─────────────────●  ◀ merge
               C1    C2    C3           C4              C7
                                (branching point)   (merge commit)

Comandos para trabajar con ramas

CREAR Y CAMBIAR DE RAMA 
# Ver en qué rama estás y qué ramas existen
git branch
# * main      ← el asterisco indica la rama actual

# Crear una rama nueva
git branch feature/nueva-seccion

# Cambiar a esa rama
git checkout feature/nueva-seccion

# Forma moderna: crear Y cambiar en un solo comando
git checkout -b feature/nueva-seccion
# O también:
git switch -c feature/nueva-seccion

# Volver a main
git checkout main
# O también:
git switch main
FUSIONAR RAMAS (MERGE) 
# Primero vuelve a la rama destino (normalmente main)
git checkout main

# Fusionar la rama de la feature con main
git merge feature/nueva-seccion

# Updating a1b2c3d..9f8e7d6
# Fast-forward
#  nueva-seccion.html | 15 +++++++++++++
#  1 file changed, 15 insertions(+)

# Eliminar la rama (ya no la necesitas)
git branch -d feature/nueva-seccion
SUBIR UNA RAMA A GITHUB 
# Subir la rama a GitHub (para que tu equipo la vea)
git push origin feature/nueva-seccion

# Ver todas las ramas (locales y remotas)
git branch -a
# * feature/nueva-seccion
#   main
#   remotes/origin/feature/nueva-seccion
#   remotes/origin/main

Pull Requests — La forma profesional de trabajar

Un Pull Request (PR) es una solicitud para que alguien revise tu código antes de fusionarlo con la rama principal. Es la base del trabajo en equipo moderno.

1. Creas una rama y trabajas en tu feature

git checkout -b feature/mi-nueva-funcion → haces commits.

2. Subes la rama a GitHub

git push origin feature/mi-nueva-funcion

3. Abres el Pull Request en GitHub

GitHub te muestra un botón "Compare & pull request". Escribes una descripción de qué cambiaste y por qué.

4. Tu equipo revisa el código

Pueden dejar comentarios, pedir cambios, o aprobar. Es la "revisión de código" (code review).

5. Se hace el merge

El responsable del proyecto acepta el PR. Tu código queda en main. La rama se puede eliminar.

🏆 Buenas prácticas de PR

Un buen PR tiene: título claro, descripción de qué se hizo y por qué, capturas de pantalla si hay cambios visuales, y es pequeño (1 PR = 1 tarea). Los PRs gigantes son difíciles de revisar y son una mala práctica.

Conflictos de merge — ¿Qué hacer?

Un conflicto ocurre cuando dos personas modificaron la misma línea del mismo archivo. Git no sabe con cuál versión quedarse y te pide que decidas:

CÓMO SE VE UN CONFLICTO 
# Cuando hay conflicto, Git modifica el archivo así:

<h1>Mi Página Web</h1>

<<<<<<< HEAD
<p>Versión que tienes TÚ en tu rama</p>
=======
<p>Versión del COMPAÑERO que quiere entrar</p>
>>>>>>> feature/rama-compañero

# Debes editar el archivo manualmente, elegir qué queda y eliminar
# las marcas <<<<<<<, =======, >>>>>>>
# Luego:
git add index.html
git commit -m "Resuelve conflicto en index.html"
💡 No entres en pánico

Los conflictos son normales en trabajo en equipo. No los rompiste. Git solo necesita que tú decidas cuál versión es la correcta. VS Code tiene una interfaz visual muy buena para resolverlos.

👥 Ejercicio 4 — En parejas
Flujo completo con Pull Request
⏱ 20 min
🎯 Objetivo: Simular el flujo real de trabajo en equipo: uno es el "owner" del repo y el otro hace un PR con una mejora.
  1. Persona A: en GitHub, ve a Settings → Collaborators y añade a Persona B Comparte el repositorio del ejercicio anterior con tu compañero.
  2. Persona B: acepta la invitación y clona el repo de A git clone URL_DEL_REPO_DE_A
  3. Persona B: crea una rama nueva git checkout -b feature/mejora-estilos
  4. Persona B: modifica el CSS o añade contenido nuevo. Hace commit git add . → git commit -m "Mejora los estilos del header"
  5. Persona B: sube la rama: git push origin feature/mejora-estilos GitHub mostrará un banner "Compare & pull request". Haz click.
  6. Persona B: abre el Pull Request con descripción Describe qué cambiaste. Asigna la revisión a Persona A.
  7. Persona A: revisa el código en GitHub y aprueba / comenta Ve a la pestaña "Files changed" y deja al menos un comentario.
  8. Persona A: hace el merge del PR Click en "Merge pull request" → "Confirm merge". ¡Listo! El código de B está en main.
