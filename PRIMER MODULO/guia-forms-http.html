01

Cliente & Servidor

Antes de tocar código, necesitas entender el modelo que mueve toda la web. Son dos actores distintos que se comunican por red.

🖥️ Cliente
tu navegador
request →
━━━━━━━━
← response
⚡ Servidor
Node, Python, etc.
query →
━━━━━━
← data
🗄️ Base de datos
01
El cliente inicia todo Tu navegador (Chrome, Firefox) es el cliente. Solo habla cuando tú haces algo: escribes una URL, haces click, envías un form.
02
Viaja una Request por internet Un paquete de texto estructurado sale de tu máquina hacia la IP del servidor. Lleva: método, ruta, headers y opcionalmente un body.
03
El servidor procesa y responde El servidor recibe la request, ejecuta lógica (busca en DB, genera HTML, valida datos) y devuelve una Response con un status code.
04
El cliente renderiza El navegador recibe la response y la muestra: puede ser HTML, JSON, una imagen, un error... lo que el servidor decidió devolver.
💡
HTTP es sin estado (stateless). El servidor no recuerda quién eres entre requests. Por eso existen las cookies, sessions y tokens JWT — para simular memoria.
02

El <form> en HTML

Un <form> es el mecanismo nativo del navegador para recolectar datos del usuario y enviarlos al servidor. Tiene dos atributos clave:

<!-- action = ¿a dónde van los datos? -->
<!-- method = ¿cómo los enviamos? -->

<form action="/registro" method="POST">

  <input type="text"  name="usuario"  placeholder="Tu nombre"  />
  <input type="email" name="correo"   placeholder="email@ejemplo.com" />
  <input type="password" name="clave" placeholder="Contraseña" />

  <button type="submit">Registrar</button>

</form>
⚠️
El atributo name es el que importa: es la clave con la que el servidor va a recibir el dato. Sin name, el campo no se envía.

¿Qué pasa cuando haces submit?

01
El navegador serializa los campos Toma cada input con su name y construye pares clave=valor: usuario=Ana&correo=ana@mail.com&clave=1234
02
GET → va en la URL · POST → va en el body Con GET verás los datos en la barra del navegador. Con POST viajan ocultos en el cuerpo de la request.
03
El servidor los recibe y procesa En Node/Express: req.body.usuario. En Python/Flask: request.form['usuario']

Tipos de input más comunes:

<input type="text"     name="nombre"   />  <!-- texto libre -->
<input type="email"    name="correo"   />  <!-- valida @ -->
<input type="password" name="clave"    />  <!-- oculta caracteres -->
<input type="number"   name="edad"     />  <!-- solo números -->
<input type="checkbox" name="acepto"  />  <!-- booleano -->
<select name="pais">
  <option value="co">Colombia</option>
</select>
<textarea name="mensaje"></textarea>    <!-- texto largo -->
03

Métodos HTTP

HTTP define verbos que indican la intención de la request. Cada uno tiene un significado semántico claro:

GET

Obtener datos

Lee información del servidor. No modifica nada. Los datos van en la URL.

POST

Crear / Enviar

Crea un nuevo recurso. Los datos van en el body, no en la URL.

PUT

Actualizar

Reemplaza un recurso completo. Necesitas enviar todos los campos.

DELETE

Eliminar

Borra un recurso del servidor. Generalmente va el ID en la URL.

🎯
Un form HTML solo soporta GET y POST nativamente. Para usar PUT o DELETE necesitas JavaScript (fetch/axios) o un campo oculto <input type="hidden" name="_method" value="PUT"> si tu framework lo soporta.

¿Cómo se ve una request GET vs POST?

GET https://api.ejemplo.com/usuarios ?nombre=Ana&pagina=2

Con GET los parámetros van en la URL después del ?. Son visibles, guardados en historial, sin body.

POST https://api.ejemplo.com/usuarios 
// Body de la request POST (no visible en la URL):
{
  "nombre": "Ana",
  "correo": "ana@mail.com",
  "clave":  "supersecreta"
}

Con POST los datos viajan en el cuerpo de la request — no aparecen en la URL, pero tampoco son secretos sin HTTPS. Siempre usa https://.

Ejemplo con fetch() en JavaScript:

// GET — leer usuarios
const res = await fetch('/api/usuarios');
const data = await res.json();

// POST — crear usuario
const res = await fetch('/api/usuarios', {
  method: 'POST',
  headers: { 'Content-Type': 'application/json' },
  body: JSON.stringify({ nombre: 'Ana', correo: 'ana@mail.com' })
});

// DELETE — eliminar usuario con id 42
const res = await fetch('/api/usuarios/42', { method: 'DELETE' });
04

Request & Response

Toda la comunicación web son mensajes de texto estructurados. Veamos exactamente qué viaja en cada dirección:

📤 REQUEST (cliente → servidor)
POST /registro HTTP/1.1 Host: miapp.com Content-Type: application/json Authorization: Bearer eyJ... Accept: application/json { "usuario": "ana", "correo": "ana@mail.com" }
📥 RESPONSE (servidor → cliente)
HTTP/1.1 201 Created Content-Type: application/json Set-Cookie: session=abc123 Cache-Control: no-store { "id": 99, "usuario": "ana", "ok": true }

Anatomía de una Request:

★
Request Line POST /registro HTTP/1.1 — el método, la ruta, y la versión del protocolo.
★
Headers Metadatos: Content-Type (qué formato envías), Authorization (tu token), Accept (qué formato quieres recibir).
★
Body (opcional) Solo en POST, PUT, PATCH. Contiene los datos en JSON, form-data, texto, etc. GET no tiene body.

Anatomía de una Response:

★
Status Line HTTP/1.1 201 Created — versión + código numérico + mensaje. El código lo dice todo.
★
Headers Content-Type (qué te mando), Set-Cookie (guardar sesión), Cache-Control (cómo cachearlo).
★
Body La respuesta en sí: JSON con datos, HTML de una página, texto de error, o nada (en un 204 No Content).
🔍
Tip práctico: Abre DevTools (F12) → pestaña Network. Cada vez que cargues una página o envíes un form, verás exactamente qué requests salieron, qué headers llevaban, y qué respondió el servidor. 
// En el servidor (Node + Express) así ves el request:
app.post('/registro', (req, res) => {
  const { usuario, correo } = req.body;    // body
  const token = req.headers.authorization;  // header
  const pagina = req.query.pagina;           // ?pagina=2 en la URL
  const id = req.params.id;                 // /usuarios/:id

  res.status(201).json({ ok: true, id: 99 });  // response
});

06

Status Codes

El servidor siempre responde con un código de 3 dígitos. El primer dígito define la categoría.

2xx

Éxito
200 OK · 201 Created · 204 No Content

3xx

Redirección
301 Moved · 302 Found · 304 Not Modified

4xx

Error del cliente
400 Bad Request · 401 Unauth · 404 Not Found

5xx

Error del servidor
500 Internal Error · 502 Bad Gateway · 503 Service Unavailable

Los más importantes que verás a diario:

200 OK            → Todo bien. La respuesta tiene el contenido pedido.
201 Created        → Se creó el recurso (respuesta típica de POST).
204 No Content      → Ok pero sin body (respuesta típica de DELETE).
301 Moved Perm.     → El recurso cambió de URL para siempre.
400 Bad Request     → Tu request tiene errores (datos inválidos, JSON roto).
401 Unauthorized    → No enviaste credenciales o son inválidas.
403 Forbidden       → Autenticado pero sin permiso.
404 Not Found       → El recurso no existe en esa ruta.
422 Unproc. Entity  → Datos con formato correcto pero lógicamente inválidos.
500 Internal Error  → El servidor explotó. Bug en tu backend.
✅
Ya sabes lo esencial. Con esto puedes entender cualquier API, debuggear problemas de red, y construir forms que realmente se comuniquen con un servidor. El siguiente paso es explorar fetch(), async/await, y cómo leer errores en DevTools Network. 
// Patrón completo: form → fetch → manejo de respuesta
async function enviarForm(e) {
  e.preventDefault();  // evita que el form recargue la página

  const datos = {
    nombre: document.getElementById('nombre').value,
    correo: document.getElementById('correo').value,
  };

  try {
    const res = await fetch('/api/usuarios', {
      method: 'POST',
      headers: { 'Content-Type': 'application/json' },
      body: JSON.stringify(datos),
    });

    if (!res.ok) throw new Error(`Error ${res.status}`);

    const json = await res.json();
    console.log('Creado:', json);

  } catch (err) {
    console.error('Falló:', err.message);
  }
}

document.getElementById('mi-form').addEventListener('submit', enviarForm);
